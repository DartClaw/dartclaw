# ADR-046: Workflow Iteration-Internals E-Track — Design Verdicts for Map/Foreach Consolidation (E1), Token Attribution (E3), and Merge-Resolve Serialization (E4)

## Status

Proposed — 2026-07-06 (targets the **0.20** workflow-simplification E-track, FR7/FR8/FR9). Design-first per the PRD's design-gated requirement: this ADR must land and be reviewed before any implementation FIS. It records **three** decisions in one record because they share a scope (the `dartclaw_workflow` iteration internals), a method (`andthen:architecture` trade-off+decompose with adversarial invariant-proof verification), and a scoring gate (invariant preservation as a hard gate; conservative disposition). **Verdicts: E1 → Keep both controllers (owner decision 2026-07-06, no implementation); E3 → Refine to coverage-only; E4 → Refine (gated implementation).** No naive candidate survived the invariant gate; E1's reopened option choice was settled by the owner to keep both controllers rather than pursue the desugar. Full per-item trade-off/decompose reports: [`design-e1-map-foreach-controller-consolidation.md`](../bundle/docs/specs/0.20/workflow-simplification-residue/design-e1-map-foreach-controller-consolidation.md), [`design-e3-foreach-token-attribution.md`](../bundle/docs/specs/0.20/workflow-simplification-residue/design-e3-foreach-token-attribution.md), [`design-e4-merge-resolve-serialization.md`](../bundle/docs/specs/0.20/workflow-simplification-residue/design-e4-merge-resolve-serialization.md).

**Related:** [ADR-041](041-framework-agnostic-workflow-engine-generic-output-validation.md) (engine `.dart` stays framework-neutral — all three changes are structural/control-flow, no skill semantics), [ADR-024](024-workflow-step-semantics.md) (engine-computed bookkeeping the cursors/budget implement), [ADR-028](028-unified-workflow-step-retry-authority.md) (retry authority the token non-charging and exhaustion paths preserve), [ADR-023](023-workflow-task-boundary.md) (workflow↔task boundary the E4 run-scoped cancel respects).

## Context

`packages/dartclaw_workflow` carries three iteration-internal mechanisms the 0.20 over-engineering audit flagged as accidental complexity: two near-duplicate iteration controllers (E1), a deferred token-attribution scheme with an evaluation-only regex scan (E3), and a cancel-and-requeue merge-resolve exhaustion path that requires a fragile live-task-id threading invariant (E4). Each was scored against owner-confirmed weighted criteria — **invariant preservation is a hard gate** (not a weight); among gate-passers, complexity reduction 40% · regression risk 30% · compat-surface stability 20% · migration cost 10% — under a **conservative** disposition (recommend Proceed only on positive proof). Each candidate's invariant proof was then attacked by two code-grounded adversarial lenses; the decisive claims were re-verified by direct file read. The candidates were treated as approaches to *validate, refine, or reject* — not decisions.

## Decision

### E1 / FR7 — Map/foreach controller consolidation → **Keep both controllers (no implementation)**

**Candidate (rejected):** desugar plain `mapOver` into a one-child `foreach` at parse time and retire `map_iteration_runner.dart` + `map_iteration_dispatcher.dart` (~992 LOC).

The design surfaced that the map and foreach controllers are **not two spellings of one behavior**; they diverge in **six** load-bearing ways: (1) aggregate element shape — the map shape is `_buildCodingResult`/single-output-unwrap/outputs-map, **not** the foreach "child-step-keyed object" the brief assumed; (2) per-iteration context-key namespace (`<mapStepId>[i]` vs `<childId>[i]`); (3) resume-cursor nodeType (`.map` vs `.foreach`); (4) dependency-readiness after a failed prerequisite (map unblocks dependents unconditionally and *settles*; foreach unblocks only on success and *pauses the run* via `_foreachDependencyHold`); (5) sub-step resume granularity (foreach's `completedSubStepIdsByIndex` skips a done child; map re-dispatches the whole item); (6) promotion/merge-resolve (map rejects the serialize-remaining sentinel; foreach runs the full merge-resolve + cancel-and-requeue path). Preserving map semantics under a desugar forces the runtime to keep a **persistent origin marker + behavior-switching conditionals** across those six surfaces — which erodes the "single node kind" complexity win the desugar was chosen for.

**Decision (owner, 2026-07-06):** the re-score gate resolves to **Option C — keep both controllers; do not pursue the parse-time desugar.** Rationale: story 30 (`56076d1b`) already extracted the *safe* shared-engine dedup into `iteration_dispatch_engine.dart`; the residual consolidation the desugar would add buys a marginal LOC reduction (map-specific logic survives as marker-gated conditionals) in exchange for a six-surface behavior-switching burden, six new regression tests, and destructive-git coupling — a poor trade against the "lightweight / smallest change / when in doubt leave it out" mandate. **No implementation FIS; FR7 ships no code.** Two follow-ups: (a) reconcile the `CLAUDE.md` "synthetic merge-resolve preflight" invariant doc drift (see Implementation Notes); (b) if a future contributor revisits consolidation, this ADR's six-surface analysis is the entry cost to price in. **E2 (settle-policy matrix) stays untouched** (DO-NOT-RE-OPEN). The `visibleSkillFor` consumer at `map_iteration_runner.dart:240` stays (it is not dead while the map controller lives).

### E3 / FR8 — Foreach token attribution → **Refine → coverage-only (High)**

**Candidate:** credit foreach-scope tokens incrementally at child-settle into the persisted `run.totalTokens`, deleting `foreachScopeConsumedTokens` + `excludeKeys` + the never-overlap invariant. **The simplification premise is falsified; reject both crediting schemes and ship the missing tests instead.**

- **Reject persisted-run crediting** — the deferred single-completion write is what makes cancelled/interrupted iterations charge **zero by absence** (the abort path returns `null` before the completion sum); incremental crediting would require active debit-on-cancel, resume re-credit suppression, and convergence de-dup — *more* coordination on the riskiest subsystem.
- **Reject the scalar-counter refinement** — it is **structurally** disqualified: `foreachScopeConsumedTokens` is called with *different* `excludeKeys` per site. The foreach check sites (`foreach_iteration_runner.dart:398/:621`) **include** the live nested-loop checkpoint; the nested-loop site (`loop_step_runner.dart:377/:381`) **excludes** it (because `loopTokens` already holds it). A scalar counter cannot include and exclude the same value — deleting `excludeKeys` double-counts and fires `maxTokens`/the 80% warning early (verified: `maxTokens=1000`, loop consumed 700=300+400 → counter 1000, kills the loop 300 tokens early). Holds even inline/sequential.
- **Record as load-bearing** (not residue): the deferred single-completion write, the never-overlap property, `excludeKeys`, and the loop-child-only token-key removal.
- **Ship the coverage gap as the FR8 deliverable** — the current suite could stay green through a regression (`workflow_budget_monitor_test.dart` passes `additionalTokens=0` everywhere; only `tokenCount==0` cancelled-direct-child is pinned). Add: (a) non-zero `additionalTokens` warning/exceeded tests with a firing-iteration assertion; (b) a non-zero-partial cancelled-**direct**-child test; (c) a nested-loop byte-equivalence test across ≥2 iterations + a checkpoint-retained resume; (d) a mid-scope un-inflated `run.totalTokens` assertion. No charging change.
- **Defer any future simplification** past 0.20, gated on all tests green + E1 landing (so a counter could live once in `iteration_dispatch_engine`) + a design preserving per-site exclusion and the per-body-step top-level-loop credits (`loop_step_runner.dart:507/584/845`).

### E4 / FR9 — Merge-resolve exhaustion serialization → **Refine (Low)**

**Candidate:** on exhaustion, stop dispatching and let in-flight siblings settle naturally, deleting `_drainAndRequeue` and the `iterTaskIds`/`onTaskCreated` threading. **Retain the let-siblings-settle direction, but gate it — the naive form and the preliminary's own phase-collapse refinement are both rejected.**

The core insight is verified: **integration-branch integrity rests on the reentrant per-project `RepoLock`, not the drain** (the drain's job is the retry *policy*), and with the drain gone the live-task-id threading (which caused a production bug) becomes genuinely removable. But the FIS is admissible only if it adopts these hard gates:
1. **Preserve a terminal "serial-attempt-consumed" marker — do NOT flatten the phase model.** `phase: drained` (sole writer `merge_resolve_coordinator.dart:723`, inside the to-be-deleted drain) is the **sole** serial-re-conflict termination signal (`attempt_counter` clears only on success); collapsing it loops the requeue forever. Relocate the terminal transition onto the serialize path.
2. **Route timeout/force-fail through a run-scoped task cancel** (by `workflowRunId`), **not** `_failRun` (verified: `workflow_executor.dart:924-940` cancels no tasks) — else a stuck sibling keeps the RepoLock + a slot. This cancel needs no live task id → compatible with removing the threading.
3. **Replace the 30s drain cap with a bounded settle-timeout; the wall-time backstop is a HARD precondition** — do not remove the cap unless the one-shot wall-time backstop (FIS story 17) has landed OR force-cancel is proven to promptly kill a wedged provider process (a task-cancel transition alone won't).
4. **Scope integrity to the CLI path; fix the server path via FR3/D2.** `runResolverAttemptUnderLock` + the whole-promote `RepoLock` are wired only on the CLI standalone path. **Verified 2026-07-06:** the server path is *not* unlocked — `WorkflowGitPortProcess` locks per-git-op and `MergeExecutor` enforces `expectedBaseSha`/clean-index guards, so concurrent promotes are caught as **detected failures, not corruption**. The gap is *optimistic* (per-op + SHA) vs the CLI's *pessimistic* (whole-promote transaction) serialization, yielding spurious promote failures + merge-resolve churn under `maxParallel>1`. Its natural fix is FR3/D2's git-adapter dedup (one owner → wrap the promote + resolver span in one lock on both paths); it is a robustness gap independent of FR9.
5. **Complete the threading removal (the FR9 payoff):** repoint the one surviving `iterTaskIds` reader (attribution at `foreach_iteration_runner.dart:278`) to a write-once first-task-id; co-remove the nested-loop `onTaskCreated` threading; confirm `grep onTaskCreated` → 0 live consumers. Drop `WorkflowSerializationEnactedEvent.drainedIterationCount` and the `mapCtx.aborted` reset carve-out only after confirming nothing relied on them.

## Consequences

**Positive**
- The design corrects the audit's misreads before any code: it surfaces E1's true six-surface compat cost (and the aggregate-shape correction), proves E3's flagged mechanism is load-bearing (preventing a wrong "simplification"), and confirms E4's direction while catching a self-wedging refinement and a factually-wrong teardown assumption.
- E3 ships a real robustness win (the missing byte-equivalence coverage) at zero charging-path risk. E4 removes an invariant that caused a production bug and surfaces a server-path robustness gap now routed to FR3/D2.
- One ADR (not three) for three internal-refactor investigations that mostly concluded "refine/hold" — proportionate to the content, matching the multi-decision house style.

**Negative**
- E1 is settled to **keep both controllers** (no code) — the desugar's marginal win didn't clear its six-surface behavior risk once story 30's shared-engine dedup was accounted for; the ~992 LOC of map controllers stay. E4 remains conditional: its terminal marker + run-scoped cancel + settle-timeout make it not a pure deletion, and it depends on FR3 (server-path lock) — but its wall-time precondition (story 17) is already met, and with E1 closed it no longer couples to E1.
- FR8 delivers no LOC reduction — the target was load-bearing.

## Alternatives Considered

- **E1 (chosen: keep both / Option C):** naive parse-time desugar rejected (gate FAIL — three regressions by construction); runtime dispatch-seam merge keeping `MapNode` rejected (partial retirement, re-opens the stories-15/30 extraction); accept-the-cursor-break rejected (a decision, not a proof). Keep-both wins on the re-score: story 30 already captured the safe shared-engine dedup, so the residual desugar win doesn't clear its six-surface behavior risk.
- **E3:** pure incremental crediting (gate FAIL); scalar scope counter (gate FAIL after adversarial verification — per-site include/exclude); hybrid live-credit + reconcile (dominated, persisted mid-scope inflation); status-quo with no coverage change (insufficient — leaves the gap).
- **E4:** pure let-settle (gate FAIL — deletes the only time bound); status-quo keep-drain (passes, delivers nothing, keeps the prod-bug invariant); keep-drain + generation-guard for the threading (gate FAIL — #4 requires the threading *removable*, and keeping the drain keeps a cancel target).

## Implementation Notes

- **FIS entry gates.** E1: none — no implementation (keep both controllers); only the `CLAUDE.md` preflight-invariant doc reconcile below. E3: the four coverage tests (test-only on production behavior). E4: the terminal-marker design + four tests (bounded-settlement force-fail, serial-re-conflict **termination**, marker-based crash-resume, force-fail-cancels-by-run-id). Keep the no-drop / completes-serially / already-promoted-untouched coverage.
- **Sequencing.** E1 is closed (no code), so E4 no longer couples to it. E4 depends on FR3 (its server-path lock parity rides FR3/D2). E3's future simplification (if any) is deferred past 0.20.
- **Framework-neutrality (ADR-041).** All three are structural/control-flow/wiring changes with zero skill-name or artifact literals.
- **Doc currency.** Reconcile the `CLAUDE.md` "synthetic merge-resolve preflight tracks dispatch generation" invariant (lists three gates; the code also gates on `!step.isForeachController`) when FR7 lands; extend the nested-loop token invariant to state `excludeKeys` is load-bearing.

## Project Compliance

Aligns with the **Core Philosophy (Binding)** — "smallest change / reuse before you build / root causes over workarounds / when in doubt leave it out." Each verdict refuses a superficially-clean deletion that would smuggle a regression (E1's six behavior changes, E3's load-bearing mechanism, E4's self-wedging phase collapse), and each makes the win *provable* before committing — including explicit exits (E1 pivot to keep-both, E3 coverage-only). Honors the DO-NOT-RE-OPEN discipline (E2 untouched) and the heightened conservative gate for E4's destructive-git surface.
