# Changelog

## 0.12.0

- **Changed (breaking)**: dual-mode skill contract collapsed to single-mode file-based. `dartclaw-prd`, `dartclaw-plan`, and `dartclaw-spec` always write artifacts to disk at the canonical `artifact_locations.*` paths and always emit paths via `contextOutputs` — the inline-emission branches are removed from each SKILL.md. Workflow steps downstream read the files via `file_read`. `dartclaw-prd` / `dartclaw-plan` additionally gain a read-existing detection branch: when `context.project_index.active_prd` / `active_plan` references an existing file, they reuse it and emit `prd_source` / `plan_source` as `"existing"` instead of re-synthesizing.
- **Changed (breaking)**: `WorkflowStep.entryGate` generalized from loop-only to all step kinds. The executor evaluates `entryGate` before `gate` in every step-kind branch (MapNode, serial, parallel group, loop-step, foreach-child). False `entryGate` skips the step cleanly (fires a new `StepSkippedEvent`, advances the cursor) rather than pausing the run. The validator accepts both bare-key (`prd_source == synthesized`) and dotted (`review.findings_count > 0`) forms. `GateEvaluator` now supports `== null` / `!= null` literal comparisons — missing keys and empty values are considered null; the literal string `"null"` also matches null. Numeric-comparison empty-→-0 fallback is preserved.
- **Added**: `gitStrategy.artifacts` sub-block (`commit: bool`, `commitMessage: string`, `project: string`). When at least one artifact-producing step is present the default `commit` is `true`; `commit: false` combined with `worktree: per-map-item` is a validator error (worktrees cannot inherit uncommitted files), and a warning for `worktree: shared`. The executor's post-step commit hook fires `git add <paths> && git commit` in the configured project's working tree for any step that produced path-shaped outputs, so per-map-item worktrees branching from the workflow branch inherit the committed artifacts.
- **Added**: `gitStrategy.worktree.externalArtifactMount` sub-block with two modes. `mode: per-story-copy` (the default, least-privilege) copies exactly one file from a sibling project's working tree into each per-map-item worktree at the same relative path — the skill's `file_read({{map.item.spec_path}})` resolves identically in both workspaces. `mode: bind-mount` is an opt-in directory mount that broadens the worktree sandbox and must be justified in the profile README. `WorktreeManager.applyExternalArtifactMount()` performs the runtime copy after worktree creation.
- **Changed**: `dartclaw-discover-project` output contract extended with `active_milestone`, `active_prd`, `active_plan`, and `artifact_locations: {prd, plan, fis_dir}`. `framework-markers.md` documents the per-framework convention table. Resolution order: prompt `MILESTONE` hint → State-document version marker → semver-highest plan directory.
- **Changed (breaking)**: shipped `plan-and-implement.yaml` updated to the new contract — `prd` / `plan` emit paths + source tags; `review-prd` and `plan-review` gain `entryGate: "*_source == synthesized"` so they skip when the upstream artifact was reused; review steps declare `allowedTools: [file_read]` and pass the artifact path through instead of rewriting the file; `plan` receives `prd_review_findings` via `contextInputs` so planning accounts for review feedback without PRD mutation; `story_specs` remains a structured array of per-story records (unchanged shape) but each record now carries `spec_path` as a path to the FIS file on disk. The four profile mirrors (`docs/testing/workflows/`, `workflows-dartclaw/`, `plain/`, `visual/`) are updated to match.
- **Added**: `OutputFormat.path` value. Treated as text at runtime; the distinct format surfaces artifact-producing intent in the workflow contract and lets the engine recognise artifact-producing outputs for the commit hook.
- Migration: authors of custom workflows referencing `dartclaw-prd` / `dartclaw-plan` / `dartclaw-spec` should update `contextOutputs` to declare `prd` / `plan` / `story_spec` with `format: path`; add `prd_source` / `plan_source` outputs when read-existing reuse is desired; add `entryGate: "<source_key> == synthesized"` to downstream review steps so they skip when artifacts were reused; set `gitStrategy.artifacts.commit: true` (or rely on the default when artifact-producing steps are present) when combining artifact-producing steps with `worktree: per-map-item`. Profiles crossing two repos should configure `gitStrategy.worktree.externalArtifactMount` with `mode: per-story-copy`.

## 0.11.0

- **Added**: SKILL.md frontmatter may declare a neutral `workflow:` block with `default_prompt` (String) and `default_outputs` (Map — same shape as a step's `outputs:`). When a workflow step references the skill and omits its own `prompt:` / `outputs:`, the engine uses the frontmatter defaults. This removes the long-standing duplication between each skill's `agents/openai.yaml` and the three built-in workflow YAMLs and drops the harness-labeled file altogether.
- **Added**: `WorkflowStep.autoFrameContext` (bool, default `true`, YAML key `auto_frame_context`). When enabled, the engine auto-appends `<key>\n{resolved value}\n</key>` blocks for every step `contextInputs` entry and workflow-level `variables:` entry that the authored prompt does not already reference (either as a literal XML tag or a `{{context.key}}` / `{{KEY}}` template reference). Tag names normalize `.` → `_` so dotted keys like `plan-review.findings_count` render as `<plan-review_findings_count>`. Auto-framing is additive — existing YAMLs already wrap context inputs, so the pass is a no-op under the shipped built-ins.
- **Added**: `WorkflowDefinitionResolver` — applies `stepDefaults` patterns, injects skill frontmatter defaults, substitutes workflow-level `{{VAR}}` bindings, and emits round-trippable YAML.
- **Added**: `dartclaw workflow show <name>` CLI. Raw by default (JSON); `--resolved` emits the fully merged YAML; `--resolved --step <id>` narrows to a single step; `--json` wraps the YAML output in a JSON envelope for scripting. Connected mode goes through `GET /api/workflows/definitions/<name>?resolve=true[&step=<id>]` (new server route); standalone mode resolves locally via the workflow registry.
- **Removed**: the 14 `packages/dartclaw_workflow/skills/*/agents/openai.yaml` files. Their `default_prompt` values moved into each SKILL.md frontmatter's `workflow:` block; `display_name`, `short_description`, and `allow_implicit_invocation` were dropped (unused by the Dart runtime).
- Migration: Dart code that consumed `SkillInfo` keeps working — `defaultPrompt` and `defaultOutputs` are new optional fields (null when the frontmatter omits the `workflow:` block). Custom workflows benefit automatically: any step that references a `dartclaw-*` skill and omits `prompt:` / `outputs:` now picks up the skill's frontmatter defaults. Author-maintained skills can opt in by adding a `workflow:` block to their own SKILL.md — no runtime registration step is required.

## 0.10.0

- **Added**: `dartclaw-prd` skill — extracts PRD creation out of `dartclaw-plan` into its own altitude. Supports the same dual-mode contract (standalone writes `prd.md` per the output-path semantics; Workflow-Step Mode returns `prd` via `contextOutputs`).
- **Changed (breaking)**: `dartclaw-plan` now requires an upstream PRD (`prd.md` in the input directory, or `prd` via `contextInputs` in Workflow-Step Mode). It fails fast with a redirect to `dartclaw-prd` when no PRD is present. It also absorbs the former `dartclaw-spec-plan` workflow (technical research, story classification, parallel FIS creation, cross-cutting review) and now emits `stories + story_specs` together in Workflow-Step Mode. A new `--skip-specs` flag produces the cheap `plan.md`-only pass; specs are generated by default.
- **Removed (breaking)**: `dartclaw-spec-plan` skill — merged into `dartclaw-plan`. All references swept from in-skill files and workflow definitions.
- **Changed (breaking)**: built-in `plan-and-implement.yaml` rewired around `discover-project → prd → review-prd → plan → story-pipeline → plan-review → remediation-loop → update-state`. The `spec-plan` step is gone, `revise-plan` is replaced by a PRD-scoped `review-prd`, and the merged `plan` step emits both `stories` and `story_specs` in one pass. The mirrored test-workspace workflow (`docs/testing/workflows/data/workspace/workflows/plan-and-implement.yaml`) stays byte-aligned with the shipped built-in.
- Migration: if you use `/dartclaw-plan <dir>` on a directory without a `prd.md`, run `/dartclaw-prd` first to synthesize one, then re-run `/dartclaw-plan`. If you invoke `dartclaw-spec-plan` directly, switch to `/dartclaw-plan` (default) or `/dartclaw-plan --skip-specs` for a plan-only pass. Custom workflow YAML referencing `dartclaw-spec-plan` must migrate to the merged `dartclaw-plan` step shape and add a `dartclaw-prd` step upstream.
