# Authoring Guidelines

Use this reference when writing a FIS.

## Principles

- State outcomes, not file edits.
- Keep each criterion testable.
- Write scenarios as observable behavior.
- Put the proof path on the page, not in your head.
- Use the project's canonical vocabulary.
- Keep the spec small enough to execute without interpretation drift.

## Good Criteria

- identify the user-visible or system-visible result
- include a verification command or observable check
- avoid vague phrasing like "works well" or "fully integrated"

## Good Scenarios

- start with the happy path
- cover key edge cases
- include failure or fallback behavior where it matters

## Good Scope Notes

- call out explicit non-goals
- note architectural constraints that should not be violated
- list the references that matter to implementation and review
